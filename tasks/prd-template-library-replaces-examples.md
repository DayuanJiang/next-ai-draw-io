# PRD: My Templates Replaces Quick Examples

## 1. Introduction

当前聊天大厅中的 `Quick Examples` 是一组硬编码示例卡片。它们主要用于演示产品能力，但对长期用户的实际工作流帮助有限，也无法沉淀用户自己的常用 prompt。

本功能将把聊天大厅中的官方示例区替换为用户自己的“我的模板库”。用户可以创建、编辑、删除、复制和使用自己的模板。模板点击后会直接发送，不再只是填入输入框。模板库仅保存在本地，并记录使用统计，帮助用户识别和优先使用高频模板。

这个功能的核心价值是把“演示入口”改造成“个人工作台入口”。

## 2. Goals

- 让用户可以用自己的模板替代官方示例作为聊天入口
- 支持模板的本地持久化，页面刷新或重新打开后仍可使用
- 支持模板的创建、编辑、删除、复制和直接发送
- 记录模板使用统计，帮助用户识别常用模板
- 保持现有聊天大厅的信息结构，避免破坏已有会话历史体验
- 为后续模板导入导出和更丰富模板能力打下基础

## 3. User Stories

### US-001: 在空聊天状态展示我的模板库
**描述：** 作为用户，我想在没有消息时看到自己的模板库，而不是官方示例，以便我能更快开始常用工作流。

**Acceptance Criteria：**
- [ ] 当聊天消息为空时，聊天大厅展示“我的模板库”而不是官方示例卡片
- [ ] 当用户没有任何模板时，展示空状态和创建入口
- [ ] 模板卡片至少展示标题、描述、标签或使用信息中的核心内容
- [ ] 文案遵循现有 i18n 机制
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-002: 在有历史会话时展示可展开的模板库区域
**描述：** 作为用户，我想在查看历史会话时仍然能快速访问模板，以便我无需离开大厅就能开始新的常用任务。

**Acceptance Criteria：**
- [ ] 当存在历史会话时，历史会话区仍然保留
- [ ] 原 `Quick Examples` 可展开区域替换为“我的模板”
- [ ] 用户可以展开或收起模板区域
- [ ] 模板区域中的交互不影响历史会话列表的选择和删除行为
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-003: 创建模板
**描述：** 作为用户，我想创建自己的模板，以便重复使用常用 prompt。

**Acceptance Criteria：**
- [ ] 用户可以手动创建模板
- [ ] 模板至少包含 `title` 和 `prompt`
- [ ] 当 `title` 为空时，系统可自动生成默认标题，规则需明确
- [ ] 空 `prompt` 不允许保存
- [ ] 模板保存后立即出现在模板列表中
- [ ] 模板数据持久化到本地存储
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-004: 编辑、删除和复制模板
**描述：** 作为用户，我想维护自己的模板库，以便持续优化和复用已有模板。

**Acceptance Criteria：**
- [ ] 用户可以编辑已有模板的标题、描述、prompt、标签和置顶状态
- [ ] 用户可以删除模板，且删除前需要确认
- [ ] 用户可以复制已有模板，并生成一个新的模板记录
- [ ] 删除模板不会影响已有聊天历史
- [ ] 编辑模板不会回写历史消息
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-005: 直接发送模板
**描述：** 作为用户，我想点击模板后直接发送，以便最快启动一次新任务。

**Acceptance Criteria：**
- [ ] 点击模板卡片会直接触发发送，而不是仅填入输入框
- [ ] 模板发送使用模板中保存的 prompt 内容
- [ ] 如果当前输入框中存在未发送内容，系统需要在发送前给出确认
- [ ] 发送模板时不破坏现有聊天发送链路
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-006: 记录模板使用统计
**描述：** 作为用户，我想知道哪些模板最常用，以便我能更快找到真正高频的模板。

**Acceptance Criteria：**
- [ ] 模板点击时 `clickCount` 增加
- [ ] 模板成功触发发送时 `runCount` 增加
- [ ] 每次发送模板后更新 `lastUsedAt`
- [ ] 模板列表支持基于统计信息排序
- [ ] 统计数据持久化到本地存储
- [ ] Typecheck/lint 通过

### US-007: 搜索、置顶和排序模板
**描述：** 作为用户，我想快速筛选和排序模板，以便在模板变多后仍能高效使用。

**Acceptance Criteria：**
- [ ] 用户可以按标题、描述或标签搜索模板
- [ ] 用户可以将模板置顶
- [ ] 默认排序优先考虑置顶模板，其次是使用频率和最近使用时间
- [ ] 当搜索无结果时，展示明确空状态
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-008: 保存当前输入为模板
**描述：** 作为用户，我想把当前输入内容直接保存为模板，以便快速沉淀高质量 prompt。

**Acceptance Criteria：**
- [ ] 用户可以从聊天输入区将当前输入保存为模板
- [ ] 保存时允许补充标题、描述和标签
- [ ] 空输入时不显示该入口或不允许保存
- [ ] 保存成功后模板立即可在模板列表中使用
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-009: 将历史用户消息另存为模板
**描述：** 作为用户，我想把一条已经验证有效的历史消息另存为模板，以便后续重复使用。

**Acceptance Criteria：**
- [ ] 用户可以从历史用户消息中触发“另存为模板”
- [ ] 保存时默认带入该消息文本作为 prompt
- [ ] 新模板保存成功后出现在模板列表中
- [ ] 仅用户消息支持该入口，助手消息不支持
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

### US-010: 导入和导出模板
**描述：** 作为用户，我想导入和导出模板，以便备份和在本地环境之间迁移。

**Acceptance Criteria：**
- [ ] 用户可以将当前模板库导出为 JSON 文件
- [ ] 用户可以从 JSON 文件导入模板
- [ ] 导入时能够处理基本校验失败并给出错误提示
- [ ] 导入策略需要明确，至少说明是覆盖、追加还是去重追加
- [ ] Typecheck/lint 通过
- [ ] 使用 agent-browser 在浏览器中验证

## 4. Functional Requirements

- FR-1: 系统必须移除聊天大厅中的官方示例卡片展示，并用“我的模板库”替代。
- FR-2: 系统必须在聊天消息为空时展示完整模板库界面。
- FR-3: 系统必须在存在历史会话时保留历史会话区域，并以可展开区域展示模板库。
- FR-4: 系统必须允许用户创建模板。
- FR-5: 模板至少必须包含以下字段：`id`、`title`、`prompt`、`createdAt`、`updatedAt`、`clickCount`、`runCount`、`lastUsedAt`、`pinned`。
- FR-6: 系统必须支持模板的可选字段，如 `description` 和 `tags`。
- FR-7: 系统必须阻止保存空 `prompt` 的模板。
- FR-8: 系统必须支持编辑模板。
- FR-9: 系统必须支持删除模板，并在删除前要求确认。
- FR-10: 系统必须支持复制模板，并生成具有新 `id` 的新模板记录。
- FR-11: 系统必须支持点击模板后直接发送模板内容。
- FR-12: 当输入框存在未发送内容时，系统必须在直接发送模板前提示确认。
- FR-13: 系统必须在点击模板时增加 `clickCount`。
- FR-14: 系统必须在模板真正触发发送时增加 `runCount`。
- FR-15: 系统必须在模板发送后更新 `lastUsedAt`。
- FR-16: 系统必须支持按标题、描述和标签搜索模板。
- FR-17: 系统必须支持模板置顶。
- FR-18: 系统必须提供默认排序逻辑，优先级为：`pinned`、`runCount`、`lastUsedAt`。
- FR-19: 系统必须支持从当前输入创建模板。
- FR-20: 系统必须支持从历史用户消息创建模板。
- FR-21: 系统必须支持将模板库导出为 JSON。
- FR-22: 系统必须支持从 JSON 导入模板，并对导入数据进行基本结构校验。
- FR-23: 模板库数据必须持久化在本地，不依赖服务器。
- FR-24: 模板库功能必须遵循现有多语言字典机制。
- FR-25: 模板功能不能破坏现有聊天发送、会话恢复和历史会话行为。

## 5. Non-Goals

- 不做云端同步
- 不做跨设备自动同步
- 不做模板级模型配置
- 不做模板绑定本地附件
- 不做模板绑定图片或其他二进制资源
- 不做团队共享模板
- 不做模板版本历史
- 不做基于模板的服务端统计分析

## 6. Design Considerations

- 模板区应沿用现有聊天大厅视觉结构，避免引入全新页面心智负担。
- 空状态需要清楚告诉用户这是“我的模板库”，并提供明确的“创建第一个模板”入口。
- 模板卡片视觉优先级应高于现在的示例卡片，强调“这是用户自己的工作资产”。
- 有历史会话时，模板区仍应易于发现，不能被埋得太深。
- 新增入口需要尽量贴近用户使用路径，优先考虑：
  - 聊天输入区的“保存为模板”
  - 历史用户消息的“另存为模板”
- 导入和导出入口建议放在模板区头部或模板管理入口附近。

## 7. Technical Considerations

- 当前官方示例入口主要位于：
  - `components/chat-example-panel.tsx`
  - `components/chat/ChatLobby.tsx`
  - `components/chat-message-display.tsx`
- 当前项目将重量级用户内容持久化到 IndexedDB，会话逻辑位于 `lib/session-storage.ts`。
- 当前 `localStorage` 更适合轻量设置项，不适合作为模板库主要存储。
- 建议模板库使用 IndexedDB 持久化，并与现有会话存储保持一致的可靠性和扩展性。
- 建议在现有数据库中新增独立的 `templates` store，而不是将模板混入 `sessions` store。
- 直接发送模板需要与现有聊天发送逻辑集成，确保与当前消息流、文件流和恢复逻辑兼容。
- 模板导入 JSON 需要版本化或最少具备结构兼容策略，避免未来 schema 扩展时难以迁移。

## 8. Success Metrics

- 用户可以在 1 次点击内直接使用模板发起聊天
- 用户可以在 30 秒内创建第一个模板
- 模板发送成功率与普通手动发送保持一致，不引入明显失败率上升
- 有模板的用户能在大厅中优先看到自己的常用模板，而不是官方示例
- 模板库在刷新页面、关闭再打开后仍然完整保留
- 搜索、置顶和排序能让 10 个以上模板的场景仍可高效使用

## 9. Open Questions

- 导入模板时的策略是否采用“去重追加”为默认行为，还是简单追加？
- 自动生成默认标题的规则是否采用“prompt 前 20 个字符”，还是按句子截断？
- 模板卡片上是否需要直接显示 `runCount`，还是只作为排序依据隐藏使用？
- “保存当前输入为模板”入口应放在输入框旁边、更多操作菜单中，还是模板区顶部？
- “另存为模板”是否只对最后一条用户消息开放，还是所有历史用户消息都开放？
